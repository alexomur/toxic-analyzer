# Toxic Analyzer Backend API Contracts

This document describes the public HTTP contract implemented by the backend MVP in `backend/`.

## Boundary

- `backend` is the public API consumed by external clients and the future `frontend`
- `model` is an internal Python HTTP service used only for inference and explainability
- `backend` translates public requests into internal `model` calls and does not expose raw upstream schemas directly

## Implemented Endpoints

- `POST /api/v1/toxicity/analyze`
- `POST /api/v1/toxicity/analyze-batch`
- `GET /api/v1/toxicity/texts/random`
- `GET /api/v1/toxicity/texts/{textId}`
- `POST /api/v1/toxicity/texts/{textId}/vote`
- `GET /health/live`
- `GET /health/ready`

The health endpoints are infrastructure endpoints. The public product contract is the toxicity API surface described below.

## `GET /api/v1/toxicity/texts/random`

### Purpose

Return one stored text for anonymous user voting.

### Response

```json
{
  "textId": "string",
  "text": "some text"
}
```

### Response Notes

- `textId` is the persistent identifier of the deduplicated row in `analysis_texts`
- random selection is weighted to prefer texts with fewer existing votes
- texts with many votes are still eligible and can still appear
- if no stored texts are available, the backend returns `404`

## `POST /api/v1/toxicity/texts/{textId}/vote`

### Purpose

Submit an anonymous user vote for a stored text.

### Request

```json
{
  "vote": "toxic"
}
```

### Request Rules

- `textId` must be a valid GUID path parameter
- `vote` is required
- allowed `vote` values: `toxic`, `nonToxic`

### Response

- `204 No Content` when the vote is accepted
- `404 Not Found` when the text does not exist

## `GET /api/v1/toxicity/texts/{textId}`

### Purpose

Return stored voting data for one text together with the latest model analytics snapshot.

### Response

```json
{
  "textId": "string",
  "text": "some text",
  "textLength": 9,
  "requestCount": 4,
  "lastLabel": 1,
  "lastToxicProbability": 0.87,
  "model": {
    "modelKey": "string",
    "modelVersion": "string"
  },
  "votesToxic": 1,
  "votesNonToxic": 2,
  "createdAt": "2026-05-02T09:00:00Z",
  "lastSeenAt": "2026-05-02T10:00:00Z"
}
```

### Response Notes

- response includes the latest stored model snapshot: `lastLabel`, `lastToxicProbability`, and `model`
- response excludes redundant derived fields such as `votesTotal`
- returns `404` when the text does not exist

## `POST /api/v1/toxicity/analyze`

### Purpose

Analyze one text for toxicity.

### Request

```json
{
  "text": "some text",
  "reportLevel": "summary"
}
```

### Request Rules

- `text` is required
- blank or whitespace-only `text` is rejected
- `reportLevel` is optional
- allowed `reportLevel` values: `summary`, `full`
- omitted `reportLevel` resolves to `summary`

### Response

```json
{
  "analysisId": "string",
  "label": 0,
  "toxicProbability": 0.12,
  "model": {
    "modelKey": "string",
    "modelVersion": "string"
  },
  "reportLevel": "summary",
  "explanation": null,
  "createdAt": "2026-04-29T12:00:00Z"
}
```

### Response Notes

- `analysisId` is generated by the backend for this response
- `label` is the normalized binary label from the model result
- `toxicProbability` is returned in the `[0, 1]` range
- `model.modelKey` and `model.modelVersion` identify the upstream model used
- `reportLevel` echoes the resolved mode
- `explanation` is always present in the schema
- for `summary`, `explanation` is `null`
- for `full`, `explanation` contains normalized explainability fields
- `createdAt` is returned as UTC

### `full` Explanation Shape

```json
{
  "calibratedProbability": 0.89,
  "adjustedProbability": 0.91,
  "threshold": 0.80,
  "features": [
    {
      "name": "some feature",
      "contribution": 0.42
    }
  ]
}
```

## `POST /api/v1/toxicity/analyze-batch`

### Purpose

Analyze a batch of texts and return item-level results plus a batch summary.

### Request

```json
{
  "items": [
    {
      "clientItemId": "optional-client-id",
      "text": "some text"
    }
  ]
}
```

### Request Rules

- `items` is required
- `items` must contain at least one element
- maximum batch size is `100`
- every item must contain `text`
- blank or whitespace-only item text is rejected
- `clientItemId`, when supplied, is treated as an opaque client value

### Response

```json
{
  "batchId": "string",
  "items": [
    {
      "clientItemId": "optional-client-id",
      "analysisId": "string",
      "label": 0,
      "toxicProbability": 0.12,
      "model": {
        "modelKey": "string",
        "modelVersion": "string"
      }
    }
  ],
  "summary": {
    "total": 1,
    "toxicCount": 0,
    "nonToxicCount": 1,
    "averageToxicProbability": 0.12
  },
  "createdAt": "2026-04-29T12:00:00Z"
}
```

### Response Rules

- response item order matches request item order
- `clientItemId` is returned unchanged
- `batchId` is generated by the backend for this response
- every item receives its own `analysisId`
- `summary.total` equals the number of accepted input items
- `summary.toxicCount` counts items with `label = 1`
- `summary.nonToxicCount` counts items with `label = 0`
- `summary.averageToxicProbability` is the arithmetic mean of item probabilities
- `createdAt` is returned as UTC

## Error Contract

The backend returns ASP.NET Core `ProblemDetails`.

Example validation response:

```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "Request validation failed.",
  "status": 400,
  "detail": "Request validation failed.",
  "instance": "/api/v1/toxicity/analyze",
  "errors": [
    {
      "field": "text",
      "message": "Text must not be blank."
    }
  ]
}
```

### Current Status Mapping

- `400 Bad Request` for validation failures
- `404 Not Found` when a requested text resource does not exist
- `503 Service Unavailable` when the internal `model` service is unavailable or returns an invalid payload
- `504 Gateway Timeout` when the internal `model` service times out
- `500 Internal Server Error` for unexpected backend failures

### Error Notes

- validation responses include `errors`
- validation responses include `detail`
- in development, unexpected failures may also include `detail`
- outside development, unexpected failures may omit `detail`

## Integration Contract Toward `model`

The backend currently relies on these internal runtime endpoints:

- `POST /v1/predict`
- `POST /v1/predict/explain`
- `POST /v1/predict/batch`
- `GET /health/ready`

The backend also expects model metadata in inference responses:

- `label`
- `toxic_probability`
- `model_key`
- `model_version`

For explain responses, the backend additionally maps:

- `calibrated_probability`
- `posthoc_adjusted_probability`
- `threshold`
- feature contribution lists

## Current Implementation Notes

- the backend captures analyzed texts asynchronously into PostgreSQL when `AnalysisCapture` is enabled
- the backend also serves anonymous random-text retrieval, text lookup, and vote registration from the same PostgreSQL `analysis_texts` table
- capture does not change the public response contract and is intentionally best-effort
- no auth layer is implemented yet
- admin and retraining endpoints are intentionally not part of the public backend MVP
